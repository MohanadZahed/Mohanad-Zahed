# Vision — Mohanad Zahed Portfolio

## Metaphor

You are the gravitational center. The tech stack circles you like planets. Scrolling pulls the ring through phases — chaos to order to dispersal — telling the story "I work with this whole ecosystem; here's how each piece fits."

It must feel calm and hypnotic, never aggressive. Soft rim lighting on the avatar, slight depth-of-field blur on the furthest logos, neutral palette with a vignette and subtle grain.

## Scroll timeline (0.00 → 1.00, normalised)

Section heights in viewport-units: Hero=1, About=1, **Manifesto=9.0** (long pinned interlude; lengthened from 6.9 for the camera-follow journey — `SECTION_VH=900`), **Skills≈2.2** (GSAP-pinned emerge intro, pulled up onto the Manifesto tail by a `-2.60` overlap margin = `0.20·SECTION_VH + 80`, then natural full-height scroll), **Knowledge=3** (pinned interlude), **Certificates=3** (sticky horizontal-scroll interlude), **Experience≈16** (sticky-stacking project cards), Contact=1 → scroll range ≈ 34.87vh. The unit in source is `svh` everywhere (see `src/sections/CLAUDE.md` → Layout rules); the `vh` shorthand in this doc is purely nominal — read it as "viewport-heights" and assume `svh` at the call site. Mobile URL-bar movement does not change page heights. Section TOPS in global progress shifted with the Manifesto→Skills rework and Skills' overlap margin makes them fuzzy — treat any global-progress number in this doc as a nominal guide and use `ScrollTrigger.refresh()` measurements for anything that needs precision.

Because Manifesto (~6.9vh) spans more than one viewport, it drives its internal choreography from a section-local progress value (computed by its own ScrollTrigger and passed to `ManifestoStage` as a prop — no store field). Global progress thresholds in scene/avatar code shift when section heights change — see "Reference scene constants" below.

Implementation note (avatar position): the avatar's world position is **not** computed from per-breakpoint world-coordinate constants. Two zero-size hidden `<div data-avatar-anchor="...">` markers are placed by CSS — one in Hero, one in About — at the exact spots the avatar should occupy. Each frame the scene reads their `getBoundingClientRect()`, projects the rect centre into world space via `rectToWorld(rect, viewport)` ([src/scene/lib/projectAnchor.ts](../src/scene/lib/projectAnchor.ts)), and lerps the anchor group's `position.x/y` between the two world points. The blend `t` is driven by the About anchor's on-screen Y — `smoothstep(vh, vh × 0.3, aboutCenterFromTop)` — so `t = 0` when the about div is at viewport bottom and `t = 1` when it has scrolled up to 30% from the bottom. `t` is published to `useScrollStore.avatarBlend` so Avatar.tsx can drive yaw from the same signal. Past `t = 1` the avatar tracks the about div exclusively; because the div is a real DOM element in normal flow, its rect scrolls upward with the page, and the avatar leaves the viewport for free as Manifesto arrives. Tuning the avatar's landing spot on a new screen size = editing the about anchor div's Tailwind classes, not Scene.tsx. The avatar mesh carries a constant `VISUAL_CENTER_OFFSET_Y = -0.9` ([src/scene/Avatar.tsx](../src/scene/Avatar.tsx)) so its torso (not its feet) sits at the anchor group's origin; the logo-ring group applies the same offset so logos stay co-centred with the avatar.

Use `smoothstep(start, end, progress)` for transitions between bands so you don't get linear-ramp artifacts.

## Reference position function

```ts
function logoPosition(
  index: number,
  total: number,
  progress: number,
  time: number,
): [number, number, number] {
  const radius = lerp(4, 1.5, smoothstep(0.2, 0.35, progress));
  const idleSpin = time * 0.08; // autonomous slow drift, rad/s
  const scrollSpin = progress * 2; // scroll layers velocity on top
  const angle = (index / total) * Math.PI * 2 + idleSpin + scrollSpin;
  const tilt = Math.sin(progress * Math.PI * 2 + index) * 0.5;
  return [Math.cos(angle) * radius, tilt, Math.sin(angle) * radius];
}
```

In the component:

```tsx
useFrame(({ clock }) => {
  const p = useScrollStore.getState().progress;
  const [x, y, z] = logoPosition(index, total, p, clock.elapsedTime);
  meshRef.current.position.lerp(new Vector3(x, y, z), 0.1);
});
```

The `0.1` lerp factor is responsible for ~40% of how good the motion feels. Keep it.

## Section storyboards

**Hero (0.00–0.04)** — Soft neutral background, vignette, grain. Avatar centered. 10 logos circle at r=4 with an autonomous slow drift (a full revolution every ~80s) so the scene feels alive even before the visitor scrolls. Each logo carries a small point light tinted to its brand color, so as it passes near the avatar it casts a colored glow on the nearest side. The hero's typographic identity is owned by `HeroLogo` ([src/sections/HeroLogo.tsx](../src/sections/HeroLogo.tsx)) — a fixed full-viewport overlay (`pointer-events-none`, `z-40`) holding a **centred** title + name + circuit underline + tagline. The name carries no background/outline/rotation; its letters are cream (`--color-secondary`) except the **"O" in "Mohanad", which is gold (`--color-quaternary`)** and anchors a circuit underline (a vertical connector from the O down to a centre node, then two horizontal segments out to end circles) — the same shape as the persistent **MOZ** brand mark.

**Hero load intro (scene-gated, typographic MOZ build).** On first load the hero *constructs* its logo under a dark veil that lifts at the end, rather than arriving all at once. The whole sequence is **gated on WebGL readiness**: a `<Preload all />` inside the `<Canvas>` forces shaders/textures to compile during the load phase, and `HeroIntroGate` ([src/App.tsx](../src/App.tsx)) watches drei's `useProgress` — once assets are loaded + compiled (then +2 rAFs to settle) it stamps `useScrollStore.heroStartedAt = performance.now()` (4 s safety-timeout fallback). **Every intro clock shares that one origin** so none starts mid-stall: the GPU init hitch lands during the dark hold *before* anything animates. Until the stamp is set, the dark veil is opaque, the construction lines are collapsed to zero length, the gold O + name letters are at opacity 0, the background SVGs are parked off-screen, and the avatar + logo ring are at opacity 0. Order once the clock starts:

1. **Construction crosshair** — four viewport-spanning lines (two horizontal, two vertical) draw out from the centre, framing a **110 px square** at screen centre.
2. **The gold "O" flickers in** inside the square (a stuttered opacity burst, then solid) — colour `--color-quaternary`.
3. **Lines + O travel** to the O's resting slot in the name. The lines stay full-viewport — each axis translates only on its cross-axis, so the crosshair stays stretched edge-to-edge as its intersection slides onto the O — and the construction O hands off to the name's real O (same gold, same metrics).
4. **The dark veil lifts** (fades to 0 over ~0.7 s) exactly as the O lands, revealing the composing hero; the crosshair lines fade out.
5. **Circuit underline draws + name letters appear** — the vertical connector drops from the O, the centre node pops, the two horizontal segments draw outward, the end circles pop; the remaining cream letters reveal **from the O outward** in step with the draw. Underline geometry is measured live in `nameWrap`-local space (`offsetLeft/Top/Width`, transform-independent) so it tracks the O through reveal, font switches, and the later collapse.
6. **Title** ("Frontend Architect") slides up; **tagline** fades in.
7. **Background SVGs fly in** ([HeroBackground.tsx](../src/sections/HeroBackground.tsx)) — re-timed to land *after* the name builds (so the construction reads against an empty dark stage); non-circuit SVGs collapse radially inward, circuit SVGs slide in per side. On completion a synthetic `resize` re-measures the cursor-proximity centres.
8. **Avatar** ([Avatar.tsx](../src/scene/Avatar.tsx)) fades in, then the **logo ring** ([LogoPlane.tsx](../src/scene/LogoPlane.tsx)) **expands out from the avatar's centre** (both kept from the prior design — they still fade in last).

All timing lives as absolute offsets (seconds from `heroStartedAt`) in [src/sections/hero.constants.ts](../src/sections/hero.constants.ts) — `HERO_LINES_*`, `HERO_O_FLICKER_*`, `HERO_O_TRAVEL_*`, `HERO_LINES_FADE_*`, `HERO_BUILD_*`, `HERO_TITLE/TAGLINE_*`, `HERO_SVG_*`, `AVATAR_FADE_*`, `LOGO_FADE_*`. The avatar fade + logo expand read `heroStartedAt` and compute `(performance.now() − heroStartedAt) / 1000` inside `useFrame` (idle motion still uses the raw R3F clock). `prefers-reduced-motion` (and any later effect re-run such as a font switch — guarded by `introPlayedRef`) skips the veil/construction and shows the hero fully composed. [src/hooks/useScrollTrigger.ts](../src/hooks/useScrollTrigger.ts) keeps `gsap.ticker.lagSmoothing(150, 33)` as a backstop so a stray >150 ms hitch resumes smoothly instead of snapping.

**Hero → corner logo (scroll-scrubbed).** As the visitor scrolls from Hero into About, `HeroLogo` collapses the full name into the persistent **MOZ** mark and parks it top-left for the rest of the site. It is driven off the **global progress store + the hero section's live `getBoundingClientRect`** (not a private ScrollTrigger), completing over `HERO_COLLAPSE_DISTANCE_FRAC` (≈ 0.45) of the section's scroll height and fully reversible. HANAD/AHED fade and close up (width → 0) so **Z gathers next to "MO"**; the underline's right side shrinks to the MOZ width; title + tagline fade. The park is **two-stage**: the mark first **rises to the top while shrinking** (`HERO_PARK_UP_*`), then **slides left** into the corner (`HERO_PARK_LEFT_*`). A dark rounded **backdrop** fades in behind the mark so the cream letters stay legible over light sections (e.g. About). The name + underline + backdrop are one transform unit (parented to `nameWrap`, measured in local space via `offset*`); the mark is decorative (`pointer-events-none`). The avatar + ring continue to slide to the About anchor independently (see About below).

**About (anchor-driven, no global-progress thresholds)** — Avatar + ring move from the Hero anchor div to the About anchor div and the avatar yaws ~36° into a 3/4 view facing the right side, where About copy reveals. The motion is driven by the About anchor's on-screen Y position (see Implementation note above): it completes when that div has scrolled to 30% from the viewport bottom. From that point the anchor tracks the About div in document space — as the user scrolls further into Manifesto/Skills/Experience/Contact, the about div scrolls up out of the viewport in normal document flow and the avatar follows it for free. It belongs to About; subsequent sections own their own visual treatment.

**Manifesto (section-local progress 0.00–1.00, ~6.9×viewport-tall)** — A scroll-pinned interlude. The 3D canvas is silent here; the entire scene is HTML/CSS-driven. The section-local progress is computed by the Manifesto's own ScrollTrigger and passed into `ManifestoStage` as a React prop (no store field).

The opening act is a **serpentine vision area** ([src/sections/manifesto/ManifestoVision.tsx](../src/sections/manifesto/ManifestoVision.tsx)). A `--color-quaternary` (warm-sand) panel fills the **left** half (a mirrored angular `clip-path` "circuit-board seam" on its right edge) and is **present from the very start** of the section — it only fades on exit. A **CI/CD pipeline** ([src/sections/manifesto/ManifestoPipeline.tsx](../src/sections/manifesto/ManifestoPipeline.tsx)) draws an **SVG serpentine S-curve**, weaving right↔left, with a circular node at each bend: `Push → Unit Test → Build Image → E2E → Deploy` (sides R/L/R/L/R — `commit` and `merge` were dropped). **The serpentine is drawn on a tall "world" canvas** (`MANIFESTO_WORLD_VH` viewport-heights tall, default 2.6) and **a "camera" follows the drawing tip** — like a snake with the camera locked on its head. The camera is a **look-at + zoom** transform on the world group (`translate(cx,cy)·scale(z)·translate(-lookX,-lookY)`, `transform-origin: 0 0`). **Intro:** the title `elevate your system` first **types in at a constant size pinned near the top edge** (`z = 1` during typing, look-at framed on `home`). Once typed, the camera does a gentle **zoom-in beat toward "system"** — a push-in that returns, `z`: `1 → INTRO_ZOOM_PEAK → 1` over `[TITLE_TYPE_END, INTRO_ZOOM_IN_END, INTRO_SETTLE_END]` — while the **dashed line drops in from above the top border** beside "system"; then the look-at **hands off to the drawing tip** (`lerp(home, tip)` over the settle window) and follows it **dead-centre in both X and Y** at `z = 1`. The framing (`home`, from `START_SCREEN_X` / `TITLE_SCREEN_Y` / `TITLE_WORLD_DY_VH`) puts the **title ~12% from the top** and the **line-start just above the top border**, so the first dashes read as coming from above the screen. The title is **not** translated away — it lives in the world beside the line-start (right-aligned, "system" at its right edge) and **pans off-screen on its own** as the look-at moves down to the tip. The tip's world position is sampled by **arc length** (`pointAtFraction` over `buildArcSamples`, matching the SVG dash reveal exactly). The line **enters from above beside "system"**, drops down, then weaves through the nodes. It is a **white dashed line** that **paints on** progressively: a solid `pathLength=1` + `strokeDashoffset` stroke is used as an SVG **mask** (sized to the full tall world, not the viewport, or it clips the path past the first screen) over the visible dashed path, so the dashes are revealed along the curve. Each node **pops** (`backOut`) **the instant the tip reaches it** — `node.drawU` is the node's **true arc-length position** on the path (computed in `computeVisionLayout`), not a hand-set guess — with a **green check-mark stamping on** (pathLength tick draw, `#22c55e`). The **three Finder boxes pop into the opposite bays** of the curve (right, left, right) as the line passes them — each carries a sky-blue border so the black box reads on the dark side too — then type their lines scroll-driven. Everything is sequenced by a single **`drawT` (0..1)** over `[VISION_BUILD_START=0.05, VISION_BUILD_END=0.30]`: a node pops at `clamp01((drawT − node.drawU)/POP_DRAW_SPAN)`, its check over `CHECK_SPAN`, each box off its own `drawU`. The geometry — path, node centres (curve amplitude `min(w·0.28, 440)`), box anchors (`±w·0.12` off-centre, pulled in to hug the line), the `anchors`/`worldW`/`worldH`, and the notebook terminus — comes from one pure `computeVisionLayout(w, h, worldH)` ([manifesto.layout.ts](../src/sections/manifesto/manifesto.layout.ts)). Every reveal (and the camera offset) is a **pure function of section progress** — no rAF/time loop, so it scrubs cleanly both ways. The **laptop is off-screen at the start**: the notebook sits at the very bottom of the tall world (the line's end-point) and only enters frame as the camera pans down to it. Once the tip reaches it (`drawT = 1`) the camera has already centred it; over `[VISION_EXIT_START=0.30, SCALE_END=0.47]` it **grows to fill the screen** (width lerp — no separate drift, since the scale-1 camera centred on the terminus already places it dead-centre) while the pipeline + boxes **fade out in place** (`CONTENT_FADE_END=0.44`) — the **sand panel persists** (fixed, not in the camera world) and is simply covered by the full-screen notebook. Because the camera is scale-1 and centred on the terminus, the grown notebook lands exactly on `computeNotebookBoxPx`, so all downstream (terminal, Skills emerge) is untouched. **Backdrop choreography** is owned by [`KnowledgeBackground`](../src/sections/knowledge/KnowledgeBackground.tsx) — a single fixed div at `-z-20` whose `background-color` is `color-mix(in srgb, var(--color-quaternary) Xpct%, transparent)` (alpha, **not** element opacity, so the disc child isn't cascaded into the fade). `Xpct` ramps `0 → 100` via `smoothstep(0.245, KNOWLEDGE_TOP_PROGRESS, useScrollStore.progress)`, so beige only paints once Skills is on stage and stays painted forever after (hidden behind the opaque disc, but reversible if the user scrolls back up). Inside the wrapper, a centred disc holds a solid-black radial gradient. The disc:

- Falls from `−120vh` → `0` translateY over `[AVATAR_SPIN_START=0.10, AVATAR_SPIN_END=0.40]` of section progress.
- Grows from `80px` → `vmax × 2.6` over `[BUBBLES_START=0.40, BUBBLES_HOLD=0.85]`.
- Lerps its `border-radius` from `50% → 0%` over `smoothstep(0.6, 1.0, expand)` so the bounding box (already larger than the viewport) fills the corners as a hard rectangle by the end of the expansion.
- Opacity is just `fadeIn = smoothstep(0.10, 0.13, p)` — **no exit fade**. Once expanded the disc stays opaque black, so the black backdrop persists into Certificates (Experience / Contact paint over it with their own content). Scrolling back up shrinks the disc out via `fadeIn` going back to `0`.

No GSAP `pin: true` needed — CSS sticky expresses the timing for free. The strip's translate is `x = lerp(0, X_visible, p)` — a single ramp that fills the entire sticky-engaged window, so there is no frozen-strip dead zone where the user scrolls but nothing moves. Below `STACK_BREAKPOINT_PX` (768 px) and under `prefers-reduced-motion`, the section degrades to a flat vertical flex-column of un-rotated cards.

Each **project card** is a `position: sticky; top: 88px;` element (≥768px viewports). Cards are siblings within the wrapper, so as the user scrolls each card pins at the top in turn and later cards stack *over* earlier ones (later siblings paint on top in document order — no z-index needed). **Cross-company hand-off:** the company wrapper itself is *not* sticky — it scrolls naturally. While inside Wrapper A, A's cards stack and pile up. When Wrapper A's bottom passes the viewport top, all of A's sticky cards release together and scroll up out of the viewport. Wrapper B (next company) then enters from below in normal document flow with its own first card, and the dance repeats.

Below the `STACK_BREAKPOINT_PX` (768 px) viewport, sticky positioning is dropped — cards become a vanilla flex-column with `gap-y` so the section remains readable as a static document on phones.

## Reference scene constants

The avatar's Hero → About motion is **decoupled** from global progress — it reads two DOM anchor divs and the about div's on-screen Y, so adding/removing sections does not require rebasing avatar thresholds. The remaining global-progress thresholds in `src/scene/lib/logoPosition.ts` (ring radius, spin, tilt) still need rebasing when page composition changes: take the document-space scroll position of an event (in vh), divide by total scroll range (in vh = total page height − 1vh). Long sections (≥ 2vh) should drive their internal choreography from a section-local progress value in `useScrollStore`, not the global one, so other sections aren't dragged through their phases as a long section grows.
