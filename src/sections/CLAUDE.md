# Section rules — scroll, layout, content

Sections are pure HTML/Tailwind that scrolls *over* the persistent 3D canvas. They do not contain any 3D code.

## Scroll source of truth

- One Zustand store: `useScrollStore` with a single number `progress: 0..1`.
- ScrollTrigger updates it on `onUpdate({ progress })`.
- The 3D scene reads it via `useScrollStore.getState().progress` inside `useFrame`.
- Section components may subscribe with a selector (re-render is fine here, just not in the canvas).
- **Never** read `window.scrollY` directly — Lenis virtualises it. **One documented exception:** scroll-*direction* show/hide of fixed UI chrome (see below).

### Scroll-direction show/hide of fixed UI chrome (`onScrollIntent`)

Fixed/overlay chrome that slides out on scroll-down and back on scroll-up — the parked **MOZ** corner mark ([HeroLogo.tsx](HeroLogo.tsx)) and the [LanguageSwitcher](../components/LanguageSwitcher.tsx) — is driven by `onScrollIntent(cb)` in [../hooks/useLenis.ts](../hooks/useLenis.ts). **This is THE mechanism for direction-based show/hide; do NOT reroute it through Lenis.**

- **Detect direction from native `window.scrollY`** on the `scroll` event, with a small px **hysteresis threshold** (so the rubber-band bounce at the tail of an iOS fling doesn't flip the direction). Fires `cb(down)` once per real direction change.
- **Why not Lenis (or raw touch input)?** Two dead ends were tried and rejected:
  - Raw `wheel`/`touchmove` finger deltas — iOS stops delivering `touchmove` once a gesture is adopted by native scrolling, so the state **latched stale** ("hides and never comes back").
  - Lenis' own `scroll` value — it's re-derived per RAF and is **non-monotonic during an iOS momentum fling**, so a threshold detector **flaps** (down/up/down…) and restarts the consumers' CSS slide transition repeatedly → the element **freezes mid-slide and only finishes when the fling stops**.
  - `window.scrollY` is **monotonic through a fling** (Lenis writes the real document `scrollTop`, so it's accurate on desktop too and `scroll` fires each Lenis RAF), so direction is unambiguous and the transition fires exactly once.
- **Compositor hint required:** the animated element needs `will-change: transform` so its slide composites on the GPU and doesn't stall on the scroll-throttled main thread during an iOS fling (`LanguageSwitcher` sets it in JSX; `HeroLogo` sets it imperatively on the parked column).

## Layout rules

- Each section: `min-h-screen`, `position: relative`. Pad with section-specific Tailwind, but don't fight the screen height.
- Canvas mount: `fixed inset-0 -z-10 pointer-events-none` in the root layout.
- Lenis owns smooth scroll. Do not also enable `scroll-behavior: smooth` in CSS.
- **Viewport-height unit: always `svh`, never `dvh` or plain `vh`.** Mobile browsers (iOS Safari, Chrome Android) show/hide the URL bar on scroll-direction reversal. `dvh` tracks that live and causes vertical jiggle as the page reflows; `vh` resolves to the *large* viewport so it disagrees with `dvh` neighbours. `svh` (small viewport — URL-bar-visible state) is stable across URL-bar toggles in every browser that supports `dvh`, so the layout never reflows. Applies to: section heights, sticky-stage heights, paddings, anchor `top`s, font-size `clamp()` fallbacks — anywhere you'd reach for a viewport-height unit. The numeric constants (`SECTION_VH`, `STAGE_SCROLL_UNIT_VH`, etc.) keep their `_VH` names; only the *unit string* appended at the call site is `svh`. For JS that needs the live viewport height, read `document.documentElement.clientHeight` (the svh-equivalent layout viewport — stable across URL-bar toggles), not `window.innerHeight` or `visualViewport.height`.

## Hero logo intro + corner mark (scene-gated, then scroll-scrubbed)

[HeroLogo.tsx](HeroLogo.tsx) is a fixed full-viewport overlay (`pointer-events-none z-40`) that owns the hero's typographic identity: a **centred** title + per-character name + circuit underline + tagline, plus the construction crosshair, the gold flicker-in "O", and a dark veil. [Hero.tsx](Hero.tsx) is now a thin wrapper (section height + `HeroBackground` + the `data-avatar-anchor="hero"` div + `<HeroLogo>`). The avatar + logo ring are **unchanged** and still fade in last. Name styling: no background/outline/rotation; cream letters except the gold "O" in "Mohanad" (`O_IDX = 1`). Full storyboard: [docs/vision.md](../../docs/vision.md) → "Hero load intro" + "Hero → corner logo". Mechanics:

- **Why gated:** the avatar GLB + logo textures compile on the Canvas's first render (~100–300 ms of blocked main thread). The whole DOM intro is held behind the dark veil until that compile is done, so the hitch lands in the dark hold instead of freezing a tween. `<Preload all />` + `HeroIntroGate` ([App.tsx](../App.tsx)) stamp `useScrollStore.heroStartedAt` once drei `useProgress` reports loaded + compiled (+2 rAFs; 4 s fallback).
- **One shared clock:** every intro layer keys off `heroStartedAt` — the `HeroLogo` GSAP timeline (lines → flicker → travel → veil-lift → underline+letters → title → tagline), the SVG fly-in ([HeroBackground.tsx](HeroBackground.tsx)), and the avatar fade + ring expand ([../scene/Avatar.tsx](../scene/Avatar.tsx), [../scene/LogoPlane.tsx](../scene/LogoPlane.tsx)). Until the stamp is set everything is hidden (veil opaque, lines zero-length, name/O opacity 0, SVGs off-screen, avatar/logos opacity 0). Idle motion still reads the raw R3F clock.
- **Imperative render-once state:** `HeroLogo` keeps animation state (`build`, `titleIn`, `taglineIn`, `collapse`) in a plain object and a single `render()` applies it to the DOM, so the timed intro and the scroll collapse never fight a shared CSS property. The intro GSAP timeline tweens those fields (`onUpdate: render`); the collapse sets `collapse` then calls `render()`. JSX style props carry **only static styling** (no opacity/transform/width) so imperative mutations survive React re-renders; the before-paint hide runs in `useLayoutEffect`.
- **Timing:** all offsets live in [hero.constants.ts](hero.constants.ts). Order: lines → O flicker → travel → veil lift → underline+letters → title → tagline → SVG fly-in → avatar → ring.
- **Scroll-scrubbed collapse → MOZ corner mark:** driven off the **global progress store + the hero section's live `getBoundingClientRect`** (read in a `useScrollStore.subscribe`), *not* a private ScrollTrigger. Completes over `HERO_COLLAPSE_DISTANCE_FRAC` of the section height. HANAD/AHED fade + collapse width so Z gathers into "MOZ", the underline right shrinks, then the mark parks **two-stage** (rise+shrink `HERO_PARK_UP_*`, then slide-left `HERO_PARK_LEFT_*`) top-left for the rest of the site, with a dark backdrop fading in behind it for legibility over light sections. Underline + park geometry is measured in `nameWrap`-local space via `offset*` (transform-independent) so the underline parks as one rigid unit with the letters.
- **Parked-mark hide/show on scroll direction:** once parked, the mark slides up on scroll-down and back on scroll-up via `onScrollIntent()` (native `window.scrollY` + hysteresis — **not** Lenis; see "Scroll-direction show/hide of fixed UI chrome" under Scroll source of truth for the full rationale). The animated column carries `will-change: transform` so it composites on the GPU during iOS flings.
- **Backstop:** [../hooks/useScrollTrigger.ts](../hooks/useScrollTrigger.ts) keeps `gsap.ticker.lagSmoothing(150, 33)`. Don't revert it to `0`.
- `prefers-reduced-motion` (and font-switch effect re-runs, guarded by `introPlayedRef`) skip the veil/construction and show the hero composed.
- **MOZ nav menu** lives in [MozNav.tsx](MozNav.tsx), split out from `HeroLogo` to keep concerns separate. `MozNav` is a `forwardRef` component that exposes `{ close(), toggle() }` via `useImperativeHandle` (exported type `MozNavHandle`). It receives `backdropRef`, `leftCircRef`, and `rightCircRef` from `HeroLogo` so it can measure the underline end-dot screen positions when positioning its circuit border on open. `HeroLogo` holds a `mozNavRef` and calls `mozNavRef.current?.toggle()` from the backdrop `onClick` and `mozNavRef.current?.close()` when `collapse < 0.9` (mark un-parks on scroll-back). The parked mark is tappable on every device (the old `!isCoarse` gate was removed). `MozNav`'s open/close effect branches on `matchMedia('(pointer: coarse)')`: **fine pointers** get the original drop-down panel below the mark; **coarse pointers (mobile)** get a **fullscreen-grow** menu — the overlay starts at the backdrop's measured rect and animates to fill the viewport (circuit border hidden, links centred via the `.moz-menu-overlay--full` CSS, an X `.moz-menu-close` top-right, Lenis paused via `getLenis().stop()` while open and resumed on close / link-tap). `BACKDROP_RADIUS` is exported from `hero.constants.ts` and shared by both files. CSS classes for the overlay (`.moz-menu-line`, `.moz-menu-dot`, `.nav-flip-link`) live in `index.css`. The coarse-pointer branch also toggles `document.documentElement.classList` `moz-menu-open` on open/close; `html.moz-menu-open #LanguageSwitcher` (in `index.css`, `!important` to beat the switcher's inline opacity) hides the language switcher while the fullscreen overlay is up so it doesn't cover the close (X) button — the switcher is a `z-50` sibling outside the `#HeroLogo z-40` overlay's stacking context, so it can't be beaten by raising the overlay's own z-index.

## Section-local progress

Sections that span more than one viewport-height (currently Manifesto, Skills, Knowledge, Certificates) must NOT drive their internal phases from the global `progress` field — adding/removing such a section would drag every other section through its phases. Pattern:

- Wire a per-section ScrollTrigger (`trigger: sectionRef.current, start: 'top top', end: 'bottom bottom'` — or `end: '+=Npx'` when you pin) and expose `self.progress` as section-local. Most write it into a dedicated store field (`skillsIntro`, `knowledgeProgress`, `certificatesProgress`); Manifesto is fine keeping it as React state passed via prop since only `ManifestoStage` reads it.
- Components inside that section subscribe to the section-local field, not the global one. Subscribe via `useScrollStore.subscribe(...)` and write to refs/`style.transform` directly when you need 60fps motion without re-renders (see `CertificateStrip`, and `Skills` writing transform/clip-path in `onUpdate`).
- Sibling sections that need to coordinate read the other section's geometry directly rather than a derived store field — e.g. Skills reuses `computeScreenRectPx()` (the notebook's deterministic full-size screen rect) to know where to emerge from, and a `SKILLS_OVERLAP_VH` negative margin to line its pin up with the manifesto's pinned tail. (The old `notebookProgress` / `notebookHandoff` store fields were removed — nothing read them.)
- Skills uses GSAP `pin` (with `pinSpacing`) for its emerge+zoom intro, then releases into normal flow so the full chip grid scrolls without being cropped. This is the one place `pin: true` is used; everywhere else CSS sticky suffices.
- Experience is the exception: it's a long section but its choreography is pure CSS sticky-stacking, so it doesn't need a section-local store field. Don't add one unless a future feature actually requires it.
- Certificates is a horizontal-scroll section. It uses CSS `position: sticky` on a 100svh inner stage (no GSAP `pin: true`) — the sticky-release moment at section.bottom = viewport.bottom is exactly the "last card centred" hand-off where the strip continues translating during the natural-flow tail.

## Skills pixel frame

`SkillsPixelFrame` ([skills/SkillsPixelFrame.tsx](skills/SkillsPixelFrame.tsx)) is a `fixed inset-0` `<canvas>` overlay that paints a dithered black pixel band hugging the Skills section as it zooms out of the laptop screen — making the bare scale feel like the edge is dissolving into pixels. Mounted only in the motion-enabled branch of [Skills.tsx](Skills.tsx) (never under `prefers-reduced-motion`). Tuning constants all live in [skills/skills.constants.ts](skills/skills.constants.ts) (`PIXEL_*`). Key design rules:

- **Scroll-driven, never time-driven.** Opacity/shimmer is a *pure function of `skillsIntro`* (specifically the zoom factor `z = smoothstep(ZOOM_START, ZOOM_END, sp)`). There is **no `requestAnimationFrame` loop** — it redraws inside `useScrollStore.subscribe(draw)`, so the pixels move only while the user actively scrolls and **freeze when scrolling stops**. Don't reintroduce a time clock.
- **Reuses the section's own geometry.** The reveal window (`winX/Y/W/H = lerp(screen-rect → full viewport, z)`) is recomputed from `computeScreenRectPx()` — the *same* math as `Skills.tsx` `apply()` — so the frame always tracks the real section edge. Draws only while `0 < z < PIXEL_DONE_Z`; clears once and stops outside that.
- **Fixed-grid anchor.** Cells are anchored to the **viewport origin** (`x = c·cell`, global integer `c/r`), NOT the moving window edge — otherwise the whole field crawls as the section grows. Each pixel holds its screen spot; the frame's inner edge sweeps across a stationary field.
- **Continuous dark→faded gradient.** Opacity *and* presence (density) both lerp from an inner to an outer endpoint along `d = max(distOutX, distOutY) / band` (0 = hugging the section, 1 = outer edge), shaped by `PIXEL_FADE_CURVE` (>1 keeps the inner band dark/dense longer). It is **not** stepped per ring and **not** per-pixel random — both of those were tried and rejected (the random one destroyed the gradient). A scattered subset shimmers within its local brightness band (`PIXEL_FLUCTUATE_*`, capped by `PIXEL_FLUCTUATE_DEPTH` so it can't break the gradient).
- **Flush to the edge, no gap.** Fills are clipped to the region *outside* the window via an even-odd path whose inner "hole" is inset by `PIXEL_EDGE_OVERLAP_PX`, so the frame paints a few px *into* the section edge (the content, painted on top, masks the overlap; at the fractional window edge it fills the hairline seam between the CSS-clipped section and the canvas). Cell rects are integer-snapped to shared edges so neighbours tile seamlessly.
- The top/bottom layer counts are intentionally asymmetric — `SKILLS_SCREEN_RECT` sits high in the viewport, so there's more room (more visible bands) below the section than above. Left as-is by design.

## i18n usage in sections

- Call `const { t, tArray, locale } = useT()` at the top of any component that renders user-visible text.
- `t('dotted.key')` for single strings; `t('key', { var: value })` for `{var}` interpolation.
- `tArray('dotted.key')` for array-valued keys (finder lines, About facts, project tasks).
- Locale-aware Typewriter: scroll-driven Typewriters update automatically when `text` changes. Auto-mode Typewriters (`<Typewriter text={t('...')} start={...} />`) must receive `key={locale}` to force remount on locale switch so the animation replays from char 0.
- See `src/store/useLocaleStore.ts` and `src/i18n/useT.ts` for the full implementation.

## Typewriter modes

`src/components/Typewriter.tsx` supports two mutually-exclusive modes — pick per call site, don't mix.

- **Auto mode (default)**: `<Typewriter text="..." start={boolean} speed={45} startDelay={120} />`. Once `start` flips true, the component types itself character-by-character on its own clock. Use this for one-shot reveals where pacing should not depend on scroll velocity.
- **Scroll-driven mode**: pass `scrollProgress` (0..1) and the displayed substring becomes a pure function of that value — `text.slice(0, ⌈p · text.length⌉)`. No internal timer; characters appear and disappear with scroll. Auto-mode props (`start`, `speed`, `startDelay`) are ignored when `scrollProgress` is defined.

Use scroll-driven mode when the typing belongs to a scroll storyboard moment and the user should feel like *they* are driving the keystrokes (current users: the manifesto section title `"elevate your system"`, the four manifesto terminal commands (`run plan` / `run build` / `run improve` / `show skills`), and both finder box panels). Map a section-local progress range `[start, end]` into `clamp01((p − start) / (end − start))` and pass that. For multi-line containers (`FinderBox`), pass a single `scrollProgress` to the box and let it slice across its lines internally — line `i` of `n` types over `[i/n, (i+1)/n]` of the box's range.

The `beat-then-hide` cursor mode resets when `done` flips back to false, so scrolling backwards out of a fully-typed range correctly re-shows the cursor.

## Accessibility

- Respect `prefers-reduced-motion`: disable ring animation, ScrollTrigger pinning, and postprocessing. The site must remain readable as a static page.
- Section landmarks: `<section aria-labelledby="...">` with a heading per section.
- Focus styles must survive Tailwind resets (don't `outline: none` without an alternative).
